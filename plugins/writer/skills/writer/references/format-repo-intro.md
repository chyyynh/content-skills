# GitHub Repo 介紹格式 — 撰寫指南

## 平台：小紅書 / 微信公眾號 / Twitter/X

社群向的 GitHub 項目推薦文，目標是讓非技術讀者也能理解項目價值，讓技術讀者想去 star。

---

## 結構模板

```
[Hook]        — 情緒切入，連結到讀者的痛點或當下熱點
[背景鋪墊]     — 1-3 句話，交代為什麼現在需要這個東西
[項目介紹]     — 一句話定位 + GitHub 基本信息（⭐、語言等）
[功能列表]     — 3-6 個核心能力，emoji 編號，每條 1-2 句
[具體例子]     — 一個 before/after 或使用場景，讓讀者「秒懂」
[個人觀點]     — 「说真的」式的主觀評價，為什麼值得關注
[局限/誠實點]  — 一個真實的缺點或限制，增加可信度
[連結]        — 🔗 GitHub URL
```

不是每個區塊都必須有，根據項目特性靈活組合。但 Hook + 功能列表 + 個人觀點 + 連結是最小集合。

---

## 各區塊寫法

### Hook

不要從「GitHub 上有個項目」開始。要從讀者的情緒、痛點、或當下熱點切入。

**有效模式**：

| 模式 | 範例 |
|------|------|
| 痛點共鳴 | 「你的经验，正在被公司"蒸馏"成AI，你知道吗？」 |
| 場景代入 | 「看视频学语言，这才是AI该有的打开方式！」 |
| 趨勢切入 | 「这两天推特上炸了——"同事.skill"火了之后...」 |
| 數字衝擊 | 「一个 3 天涨了 8000⭐ 的项目，它到底解决了什么？」 |

### 背景鋪墊

建立「為什麼現在」的語境。可以是行業趨勢、社群事件、或個人遭遇。讓讀者覺得「確實需要」再介紹項目。

### 功能列表

核心區塊。用 emoji 編號（1️⃣ 2️⃣ 3️⃣ 或主題 emoji 🛡️ 🎬 💡），每條：
- **第一句**：這個功能做什麼（動詞開頭）
- **第二句**（可選）：為什麼這很重要，或技術亮點

```
1️⃣ AI实时生成字幕——基于Whisper，任何视频/音频都能自动出字幕，不需要字幕文件
2️⃣ 双语字幕同屏显示，文本+图片格式都支持
3️⃣ 实时翻译——接Google、DeepL、Ollama、OpenAI等多个引擎
```

**原則**：
- 用讀者能理解的語言，不是 README 的技術術語
- 每條有具體性（「接 Google、DeepL」比「支持多種翻譯引擎」好）
- 控制在 3-6 條，多了讀者會跳過

### 具體例子

讓抽象功能變得可感知。最有效的格式：

**Before → After**：
```
"Redis的key必须设TTL，否则PR会被打回"
→ 蒸馏后变成 "缓存遵循团队规范"
```

**使用場景**：
```
拿来刷美剧学英语、看日综学日语，比任何语言学习App都实在。
```

### 個人觀點

用「说真的」「说实话」「我觉得」帶出主觀評價。這是整篇文章的靈魂——不是客觀評測，是一個人真心推薦。

要回答的問題：這個項目為什麼值得讀者的時間？它戳中了什麼痛點？

### 局限 / 誠實點

一個真實的缺點能大幅提升可信度。常見寫法：

- 「唯一的遗憾——目前只支持 Windows。」
- 「目前还比较早期，文档不太完善。」
- 「免費版有速率限制，重度使用需要自備 API key。」

---

## 排版要點

- **短段落**：每段不超過 3 句，大量換行
- **口語化**：寫完念一遍，念起來不順就改
- **Emoji**：功能列表用編號 emoji，段落分隔用主題 emoji，不要滿篇都是
- **具體數字**：⭐ 數、百分比、速度對比——比形容詞有說服力
- **專有名詞保留原文**：Whisper、RISC-V、yt-dlp 不要翻譯

---

## 常見陷阱

- ❌ 照搬 README 的功能列表 → 要翻譯成讀者語言
- ❌ 全篇吹捧沒有缺點 → 讀者會覺得是廣告
- ❌ 技術細節堆砌 → 非技術讀者直接關掉
- ❌ 開頭「GitHub 上有一個很好的項目」→ 沒有任何吸引力
- ❌ 沒有個人觀點，純客觀搬運 → 失去社群傳播力

```markdown
## Repo 介紹：[項目名]

**項目**：[owner/repo] — ⭐ [數量] — [一句話定位]

### 正文

[完整推薦文]

### 發布建議
- Hashtag：#標籤1 #標籤2
- 配圖建議：[截圖、demo GIF、架構圖等]
```
